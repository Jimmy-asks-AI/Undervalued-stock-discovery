# GitHub 开源量化交易系统调查与本项目建议

调查日期：2026-07-10

## 结论

你的理解基本正确：量化系统在做实时决策时不应读取未来数据。历史回测完全可以用于开发和初步验证策略；回测中的 `forward_return_20d` 等字段只是事后评价标签，不能进入当时的策略输入。

真正的问题不是“能不能使用历史数据”，而是如何保证每个历史时点只能看到当时已经公开的数据，以及如何避免在同一段历史上反复调参后把偶然结果误认为规律。

GitHub 上存在大量开源量化系统，但开源内容主要是：

- 数据管理与特征计算框架；
- 回测、账户、组合和风险引擎；
- 模拟盘与实盘连接器；
- 示例策略和研究工作流。

长期有效的真实 alpha、专有数据和生产参数通常不会完整公开。这是本次仓库调查得到的总体判断，不应把项目 star 数或示例净值当成收益证明。

## 什么叫“不使用未来数据”

正确历史回测的时间链是：

```text
历史时点 t 可获得的数据
  -> 计算特征
  -> 生成信号
  -> t+1 或预先规定的时点成交
  -> 用 t+1 到 t+H 的实际历史收益评价信号
```

其中最后一步的收益对历史时点 t 来说是“未来”，但对今天来说已经发生。它只能由评价器读取，策略函数不能读取。

例如回放 2018-10-18：

- 特征只能使用 2018-10-18 及以前可获得的数据；
- 如果财务或估值数据在 2018-10-19 才发布，就不能用于 2018-10-18；
- 信号在 2018-10-18 收盘后产生；
- 交易按规则在下一交易日执行；
- 2018-10-19 至之后 20 个交易日的收益只用于评价。

因此，标签里存在未来收益不等于策略偷看未来。真正的未来函数是把该收益、全样本分位、后来修订的数据或事后行业成分放进历史信号。

## 为什么成熟团队还要做样本外和模拟盘

历史回测足以建设系统，但不能单独保证实盘有效。主要原因是：

- 同一段历史被反复查看后，已经不再是“未见数据”；
- 参数可能只适合某几轮牛熊周期；
- 历史数据可能忽略停牌、涨跌停、成交容量和真实滑点；
- 数据源、指数编制和交易制度会变化；
- 代码在回测和实时运行中可能使用不同的数据时钟。

主流实践通常是：

```text
研究集 -> 验证集 -> 完全锁定的测试集
       -> 滚动 walk-forward
       -> 模拟盘 / dry-run
       -> 小规模人工监督运行
       -> 风控通过后再考虑自动化
```

真实前推不是运行策略所必需的数据，而是防止历史过拟合的最终审计证据。你的系统可以立即用历史数据重建，不需要等未来日期才能继续研究；只是不能把历史反复调优结果直接称为已验证实盘能力。

## GitHub 开源系统调查

### 1. qteasy

仓库：[shepherdpp/qteasy](https://github.com/shepherdpp/qteasy)

定位：面向国内市场的本地化数据、策略、回测、优化和模拟交易工具包。

值得借鉴：

- 每一步只向策略注入当时可见的数据窗口；
- 同一套策略逻辑用于回测和模拟运行；
- 支持 T+1、费用、最小交易单位和数据周期控制；
- 默认使用 Tushare 获取 A 股、指数和基金数据；
- BSD-3-Clause 许可。

局限：项目仍说明处于测试阶段；README 当前声明 Python 版本范围为 3.9 至 3.12，不能直接装进本工作区当前 Python 3.14 环境。

适合本项目：最值得借鉴其“逐时点数据窗口”和国内交易规则，也可在独立 Python 3.11 环境中作为第二回测引擎。

### 2. RQAlpha

仓库：[ricequant/rqalpha](https://github.com/ricequant/rqalpha)

定位：可扩展的 Python 算法回测与交易框架，覆盖数据、回测、模拟和分析，面向中国证券市场的适配较成熟。

值得借鉴：

- A 股交易日历和证券规则；
- 事件驱动的策略执行；
- 回测、模拟盘和分析模块分离；
- 数据 bundle 与引擎解耦。

局限：官方 README 明确限制为非商业使用；申万二级行业指数和自有 PIT 估值仍需自建数据适配器。

适合本项目：作为独立复核引擎，验证同一信号在另一套成交、费用和账户实现下是否得到一致结果。

### 3. Microsoft Qlib

仓库：[microsoft/qlib](https://github.com/microsoft/qlib)

定位：面向量化研究和机器学习的完整工作流，包含数据层、模型、组合、回测、实验记录、在线服务和 PIT 数据设计。

值得借鉴：

- 数据、模型、组合和评价松耦合；
- 明确区分 train、validation、test 时间段；
- 支持 PIT 数据库与实验记录；
- 回测同时报告基准、成本前后超额收益；
- 中国市场默认交易单位、涨跌停、费用和最小佣金等配置可建模。

局限：框架较重，主要针对股票和机器学习研究；官方说明公开示例数据暂时不可用，社区数据源质量也需要自行验证。

适合本项目：借鉴 PIT、实验记录和组合评价设计。当前只有 131 个行业，不值得整体迁移到 Qlib。

### 4. VeighNa / vn.py

仓库：[vnpy/vnpy](https://github.com/vnpy/vnpy)

定位：国内使用广泛的量化交易平台开发框架，重点是行情、交易接口、组合策略、模拟账户、数据记录和前端风控。

值得借鉴：

- 支持 XTP、TORA、东方财富 EMT 等国内证券接口；
- 组合策略模块同时支持历史回测和实盘运行；
- 包含 paper account、data recorder、risk manager；
- 适合把研究信号接到模拟盘或券商接口。

局限：券商接口通常需要相应账户、终端或授权；它不是申万行业估值研究工具，数据和信号仍要由本系统提供。

适合本项目：只在研究规则稳定后作为模拟盘和交易执行层，不应现在就引入研究主线。

### 5. QUANTAXIS

仓库：[yutiansut/QUANTAXIS](https://github.com/yutiansut/QUANTAXIS)

定位：本地股票、期货和期权的数据、回测、模拟、交易、可视化和多账户解决方案，MIT 许可。

值得借鉴：完整本地数据和账户体系、任务调度、回测和多账户架构。

局限：系统规模很大，包含 MongoDB、消息队列、Web 和 Rust 等生态；当前主版本仍带 alpha 标记。对日频行业策略而言引入成本明显高于收益。

适合本项目：参考账户和数据服务设计，不建议整体依赖。

### 6. QuantConnect LEAN

仓库：[QuantConnect/Lean](https://github.com/QuantConnect/Lean)

定位：模块化、事件驱动、支持研究、回测、优化和实盘的专业交易引擎，主要使用 C#，也支持 Python。

值得借鉴：研究、回测和实时引擎的统一事件语义；可插拔数据、交易、结果和实时处理模块。

局限：A 股数据、券商接口和申万行业数据不是开箱即用；Docker、.NET 和数据适配成本较高。

适合本项目：作为成熟架构参考，不适合作为当前最小实现。

### 7. NautilusTrader

仓库：[nautechsystems/nautilus_trader](https://github.com/nautechsystems/nautilus_trader)

定位：Rust 核心、Python 控制面的确定性事件驱动交易引擎，研究和实盘使用同一时间与执行语义。

值得借鉴：消除研究引擎和生产引擎之间的行为差异。

局限：偏专业交易基础设施，没有现成 A 股数据和券商适配；对当前日频行业系统明显过重。

适合本项目：参考时间模型，不引入依赖。

### 8. vectorbt、Backtrader 和 bt

仓库：[vectorbt](https://github.com/polakowo/vectorbt)、[Backtrader](https://github.com/mementum/backtrader)、[bt](https://github.com/pmorissette/bt)

定位：通用策略研究和回测库。

- vectorbt 适合快速批量试验和多资产矩阵计算，但越容易跑数千组参数，越需要控制过拟合。
- Backtrader 是经典事件驱动框架，但国内规则和数据需要自行实现，主仓库维护节奏较慢。
- bt 适合资产配置和可组合的组合算法，不是完整 A 股实盘系统。

适合本项目：可用于独立计算复核，但没有必要替换现有日频研究代码。

### 9. Freqtrade

仓库：[freqtrade/freqtrade](https://github.com/freqtrade/freqtrade)

定位：加密货币机器人，不适用于 A 股。

值得借鉴的是工作流而不是代码：项目同时提供 backtesting、lookahead-analysis、dry-run 和 live trade，并明确建议先运行 dry-run。这个顺序正是普通量化系统从历史研究走向真实运行的方式。

### 10. ZVT

仓库：[zvtvz/zvt](https://github.com/zvtvz/zvt)

定位：模块化量化框架，覆盖数据、基本面、技术分析和交易研究，MIT 许可。

局限：项目 README 明确不保证向后兼容，部分能力可能随作者研究方向变化。

适合本项目：参考模块组织，不作为当前主依赖。

## 对本项目的框架选择

不建议整体迁移到任何一个大型开源系统。当前最小且合理的组合是：

| 层级 | 建议 |
| --- | --- |
| 数据与 PIT | 保留现有本地缓存和 `available_date`，借鉴 Qlib |
| 策略与历史回测 | 保留精简 Python 主线，采用 qteasy 式逐时点窗口注入 |
| 独立结果复核 | 在 Python 3.11 独立环境用 qteasy 或 RQAlpha 重放同一策略 |
| 参数研究 | 可用 pandas/vectorbt 辅助，但必须记录全部试验并锁定测试集 |
| 模拟盘 | 规则稳定后接 vn.py paper account 或券商模拟环境 |
| 实盘执行 | 最后才接 vn.py 券商 gateway，并保留人工确认和前端风控 |

这比将现有系统改造成 Qlib、LEAN 或 QUANTAXIS 更简单，也更适合当前日频行业研究目标。

## 最小历史数据需求

仅使用历史数据，就足够重建“低估状态 -> 超跌修复 -> 强行业选择”的研究系统。

### 必需数据

| 数据集 | 必要字段 | 用途 |
| --- | --- | --- |
| 交易日历 | `trade_date`、是否交易日、下一交易日 | 信号和执行对齐 |
| 申万二级行业行情 | 行业代码、日期、OHLC、成交量、成交额 | 超跌、回撤、波动、流动性和未来收益标签 |
| 申万行业估值历史 | 估值日期、`available_date`、PE、PB、股息率 | 历史低估状态和行业相对估值 |
| 行业层级映射 | 二级行业、一级父行业、生效日期 | 父行业中性和口径变化审计 |
| A 股市场基准 | 中证全指或沪深300日行情 | 判断市场反弹和计算时机收益 |
| 行业选择基准 | 同日全部可用行业的等权收益 | 区分市场 beta 和行业选择 alpha |
| 现金与成本假设 | 现金收益、佣金、滑点、换手成本 | 端到端净值 |

### 强烈建议的数据

- PIT 行业成分股和成分股行情，用于构造行业上涨家数、创新低比例和成交扩散；
- 每日行业资金流快照；
- 行业盈利预测或分析师一致预期修订；
- 行业盈利、ROE、现金流和资产负债率的 PIT 聚合数据；
- 指数编制和行业分类调整记录。

### 实盘阶段才需要的数据

- 可交易 ETF 或股票篮子的实时行情；
- 成交额、买卖价差、折溢价和跟踪误差；
- 券商持仓、资金、订单、成交和撤单回报；
- 涨跌停、停牌、最小交易单位和风控限制。

行业指数本身不能直接成交。因此研究可以完全针对行业指数，但真正执行时最终仍需要 ETF 或股票篮子作为载体；这一步可以推迟到策略通过历史验证之后。

## 当前工作区已有和缺少的数据

已有：

- 131 个申万二级行业的历史行情；
- 申万二级历史 PE、PB、股息率；
- 行业成交额和价格衍生广度；
- 若干日的行业资金流快照；
- 交易成本、延迟执行、独立事件和相对收益评价代码。

主要缺口：

- 数据自动更新和新鲜度不足；
- 市场时机仍主要使用合成行业等权市场，不是稳定的可交易基准；
- 历史行业成分与分类生效日期不完整；
- 资金流 PIT 历史过短；
- 没有一段真正未参与历次调参的历史测试集；
- 没有第二回测引擎复核成交和账户结果。

这些缺口不妨碍立即重新做历史回测，但会限制结论可信度。

## Codex 可以具体做什么

### 第一阶段：历史回测重建

1. 把原始目标固定为唯一规则链：低估门禁、超跌门禁、止跌门禁、行业排序。
2. 分离 `feature_panel` 和 `label_panel`，策略代码禁止导入任何 `forward_*` 字段。
3. 为估值、财务和分类数据统一实现 `available_date <= signal_date`。
4. 采用逐交易日回放：当日收盘生成信号，下一交易日执行。
5. 用市场 timing-only、行业等权和行业 TopN 三条净值分解 beta 与 alpha。
6. 固定训练、验证和测试协议，记录所有试验，不再按版本结果继续手工追参数。

### 第二阶段：独立复核

1. 建立 Python 3.11 独立环境。
2. 将相同历史数据和冻结规则接入 qteasy 或 RQAlpha。
3. 对比信号日期、成交日期、持仓、费用、净值和最大回撤。
4. 任一事件不一致时生成逐笔差异报告。

### 第三阶段：研究自动化

Codex 可以持续完成：

- 自动下载、清洗、缓存和审计数据；
- 检测未来函数、错误 shift 和同日成交；
- 运行 nested walk-forward、bootstrap、参数扰动和分市场状态检验；
- 生成中文报告、候选行业表和 Dashboard；
- 保存每次实验的配置、数据指纹和代码指纹；
- 让多 Agent 分别负责数据、估值、窗口、行业排序和验证否决。

### 第四阶段：模拟盘和实盘辅助

规则通过后，Codex 可以：

- 接入 vn.py 模拟账户或券商模拟接口；
- 每日收盘后刷新数据并生成下一交易日观察计划；
- 对候选载体做流动性、折溢价和跟踪误差检查；
- 维护订单、持仓、退出和风险日志；
- 继续保持人工确认，直到生产门禁通过。

Codex 不能替代数据授权、券商接口权限，也不能保证策略赚钱。它最有价值的作用是把研究假设变成可复现代码，持续检查时间边界，并让同一规则在历史、模拟和真实运行中保持一致。

## 推荐的下一步

不要继续等待当前 V5.25 四条观察来决定系统方向，也不要立刻引入庞大交易框架。

下一步应建立一个新的历史回测基线，但不继续沿用版本号堆叠：

```text
PIT 历史数据
-> 每日 as-of 回放
-> 低估 + 超跌 + 止跌三重门禁
-> 窗口内行业排序
-> 下一交易日执行
-> 20/60 日收益评价
-> timing beta / selection alpha 分解
-> qteasy 或 RQAlpha 独立复核
```

这条路线完全可以立即使用现有历史数据开始，不依赖尚未发生的市场数据。未来前推只作为最终验证，不再作为继续开发的阻断条件。
