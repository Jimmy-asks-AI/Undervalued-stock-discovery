## 一、结论先行

这个项目已经是一套能运行、会自我否决、证据边界相对诚实的 A 股行业研究与 ETF 人工辅助框架；但它还不是一个已经验证有效的“低估抄底并选出强反弹行业”系统，更不能作为买卖建议或自动交易系统。

截至 2026-07-16，当前主线的正确动作是 `NO_ACTION`。数据新鲜度已通过，但择时稳健性、强行业选择、目标证据、账户状态和六角色确定性否决链均未通过。项目自己给出的 `production_ready=false`、`auto_execution_allowed=false` 是正确的，不能绕开。

综合成熟度判断：

| 维度 | 判断 | 说明 |
| --- | --- | --- |
| 行业研究底座 | 可用 | 131 个申万二级行业的价格、成交额、估值与缓存体系已经形成 |
| 低估识别 | 部分可用 | 有 PIT 意识和历史估值，但“低估”尚未成为当前 V4.70 入场的不可绕过条件 |

### 4.2 V4.70 只能算历史择时候选

V4.70 的历史统计不差：57 个事件，成本后平均收益约 3.24%，胜率约 63.16%。但随后审计显示：参数扰动通过率只有 33.33%，按 60 日冷却合并后只剩 12 个独立簇，`production_ready=false`。

更重要的是，V4.70 的冻结条件主要是负广度、上涨扩散、流动性修复、短期市场收益和波动状态，没有把 PE、PB、股息率或估值历史分位写成硬入场条件。它证明的是“某类下跌后修复窗口在历史上表现较好”，不是“系统已经识别低估抄底时机”。

因此 README 中“有效反弹窗口、100 分”的说法必须始终和“历史候选、参数脆弱、等待前推”同时出现，不能单独传播。

### 4.3 强行业选择没有通过

- 平均相对收益约 2.03%；

- bootstrap 5% 下界约 27.81%，低于 30% 门槛；
- 家族级 Bonferroni 通过数为 0；
- 历史登记状态是 `post_hoc_historical_inventory`，不是事前预注册。

V5.20 以后继续使用本地价格、估值、成交额等历史字段，也没有得到完整门槛通过的规则。当前严格结论仍是：本地历史特征不足以证明稳定的强反弹行业 Alpha。

### 4.4 前推证据才刚开始

2026-07-12 起，V5.04 的两条冻结规则登记为 `preregistered_forward_only`，实验账本有 SHA-256 前向哈希链。到 2026-07-16：

这条前推链是目前最值得保留的研究治理资产。正确做法是等新样本，不是继续从旧历史里挖新阈值。

## 五、主要问题与优先级

### P0：阻止投资使用的硬问题

#### 1. 原始目标仍未完成

“低估状态 → 超跌企稳窗口 → 强行业选择 → ETF 端到端实施”四层没有同时成立。当前只有行业数据底座和一个脆弱的历史窗口候选，强行业选择及端到端 Alpha 均未验证。

影响：任何买入、卖出、仓位或收益承诺都缺乏完整证据。

处理：继续维持 `research_only`、`NO_ACTION/WATCH`；不把 V4.70 评价分、4 条资金流观察或 ETF 映射覆盖率包装成策略有效性。

#### 2. 生产级实施数据与账户输入不完整

历史 Level-1 盘口、IOPV 和真实价差缺失；账户状态停在 2026-07-13。当前 57 笔日频成交回放是有价值的实施基线，但不能替代盘前/盘中可成交性审计。

影响：即使未来研究门禁通过，也不能直接生成个性化份额和可执行建议。

处理：只有取得可审计授权数据、更新当日账户和执行快照后，才能解除相应门禁。

#### 3. V5.25—V5.35 的“前推”链允许事后补录，不能作为未来晋级证据

当前 4 条资金流观察均为 `exploratory_fund_flow_only`，V5.28 会把它们过滤掉，所以目前没有形成虚假晋级；但这条链的时间完整性设计存在 P0 漏洞：

- [V5.25](../scripts/build_v5_25_fund_flow_forward_observer.py) 没有 `--as-of-date`，也不检查 `recorded_at` 是否早于计划入场；入退场日还使用普通工作日 `BDay`，不是 A 股交易日历；
- [V5.33](../scripts/build_v5_33_fund_flow_entry_price_freeze.py) 和 [V5.34](../scripts/build_v5_34_fund_flow_benchmark_entry_freeze.py) 可以在入场日之后，从历史行情补取首个入场价并标为 `frozen`；
- [V5.30](../scripts/audit_v5_30_fund_flow_forward_ledger_integrity.py) 不校验观察记录时间和冻结生成时间，无法证明“结果未知时已经登记”；
- [V5.27](../scripts/settle_v5_27_fund_flow_forward_samples.py) 只要冻结基准非空即可结算，没有在写回前强制执行 V5.30/V5.34 规定的至少 100 个基准行业；
- [V5.31](../scripts/build_v5_31_fund_flow_evidence_freeze_manifest.py) 在基线缺失或使用 `--update-baseline` 时，会先用当前状态覆盖基线，再比较为零变化；
- [V5.28](../scripts/build_v5_28_fund_flow_promotion_evaluator.py) 只看结算指标和冻结状态字符串，门槛通过后会直接把 `can_claim_strong_rebound_industries` 置为真。

本次已用纯内存反例复现：观察和冻结均发生在计划入场后，V5.30 仍可六项全通过、零违规。

影响：未来即使累计 30 批“合格”样本，也不能证明它们是真正的前推样本。

处理：在修复前，V5.28 必须禁止产生目标完成声明。观察账本改成追加式哈希链；强制 `detected_at < entry_cutoff`；入场后补录只能标记 `late_backfill_excluded`；冻结记录必须绑定 UTC 时间、源文件指纹、Git commit 和操作者；V5.27 写回前必须硬检查完整基准覆盖；V5.31 基线缺失应失败，更新必须新建 cohort，不能覆盖旧基线。

### P1：影响可信度、复现和维护的高优先问题

#### 4. “23/23 工程通过”和“12/12 回归通过”被表述得过重

统一回归只是依次调用 12 个脚本的 `--self-check`。完成度审计中多项实现检查依赖源码是否包含特定字符串、某个输出字段是否存在，或历史摘要是否为某个值，而不是在隔离环境中运行完整行为测试。

例如 [完成度审计脚本](../scripts/audit_etf_assisted_trading_completion.py) 会用源码字符串判断原子写入、前推边界、刷新链、Dashboard 和持仓接口是否存在；Git 门禁甚至只检查 `.git/HEAD` 是否存在。

影响：23/23 更接近“接口与审计合同存在”，不是“所有核心路径已被行为级测试覆盖”。

处理：把指标改名为“契约存在性检查”；另建真正的测试层，至少覆盖失败网络响应、陈旧/乱序数据、重复前推样本、缺失交易日、停牌/涨跌停、账户持仓和买入后风险复算。

#### 5. Git 边界审计存在明显假阳性

当前只有 33 个文件被 Git 跟踪；工作树另有 442 个未跟踪文件和 2 个已跟踪修改。未跟踪项包括 188 个脚本、121 个 `strategy_lab` 文件、100 个配置、13 个日志等。项目没有可提交的 `.gitignore`，仅在本机 `.git/info/exclude` 中排除部分输出和缓存。

现有“Git提交边界”审计只判断 `.git/HEAD` 是否存在，因此任何初始化过 Git 的目录都会通过。

影响：换机、克隆、回滚和审计复现会丢失大量历史研究资产；本机状态无法由提交记录完整重建。

处理：明确区分“当前生产主线仓库”和“历史研究档案”。把主线所需脚本、配置、schema、锁文件和关键研究登记纳入 Git；提交 `.gitignore`；对历史大文件建立只读归档清单和哈希，不要继续依赖本机未跟踪目录。

#### 6. Python 依赖不可复现

项目没有 `pyproject.toml`、`requirements.txt`、`uv.lock`、`poetry.lock` 或 Conda 环境文件。README 只写了 Backtrader 安装，但当前脚本广泛依赖 pandas、numpy、akshare、requests、openpyxl、pdfplumber 等。

本机可运行不代表新环境可运行，也无法确认未来 akshare/pandas 行为变化会不会改变结果。

处理：为当前主线建立最小依赖锁和 Python 版本声明；把历史可选依赖与当前运行依赖分开。

#### 7. 文档、变更日志和真实主线不同步

- README 顶部已更新到 2026-07-16 的当前主线，但历史章节只写到 V5.31；
- `logs/version_changelog.md` 最后停在 V2.8、2026-06-14；
- `logs/research_log.md` 最后停在 2026-06-14；
- `logs/review_correction_log.md` 最后停在 2026-06-12；
- README 同时突出 V4.70 “100 分”和后续 `production_ready=false`，容易让读者截取前者误判成熟度；
- 部分审查文字仍写“9 步刷新链”；审查时代码已经是 10 步，2026-07-18 状态整改后统一为十一项输入刷新链。

影响：项目的真实状态需要依赖多个输出报告拼接，README 和日志不能单独承担交接职责。

处理：建立一页 `CURRENT_STATUS.md`，只保留“当前目标、当前冻结规则、数据截止、门禁、下一允许动作、禁止动作”；历史版本移入索引，不再把数十个版本叙事堆在 README 主体。

#### 8. 版本化脚本过多，主线与档案边界仍不够清楚

205 个脚本中 177 个带版本号，多个脚本超过 1,000 行；旧版本大量复制逻辑。虽然当前已有统一 runner，但读者仍可能误跑旧脚本，修复也容易只落在某一版本。

影响：维护成本高，跨版本修复容易遗漏；历史实验的代码差异难以追踪。

处理：冻结历史脚本为只读档案；把当前主线抽成可测试模块，版本差异尽量落到配置和实验登记，而不是继续复制 runner。

#### 9. 六角色确定性否决链需要准确定义

当前六角色确定性否决链确实会运行，并能分别给出数据、市场、行业、ETF、账户和独立验证结果；它们本质上是确定性校验与编排模块。97 个活动 task brief 中，94 个由同一个 `industry_factor_validation_auditor` 挂名，旧的角色文档还混有个股会计质量、股东回报、ST/退市等早期职责。

影响：名称容易让人高估独立研究者之间的制衡程度。

处理：对外统一称“六角色确定性否决链”；只有真正独立输入、独立模型或独立审查过程存在时，再强调多主体协作。

#### 10. PIT 估值与行业历史宇宙仍有口径缺口

数据合同要求历史行以真实 `available_date` 为硬边界；但申万历史估值源仍缺发布时间/release-lag 审计，V5.11 使用原始估值 `trade_date` 与信号日等值合并，不能证明同日数据在决策时已经可见。

此外，回测横截面从 2015 年约 68 个行业，跃升到 2022 年约 124 个、2023 年末 131 个。2022 年既接近行业分类宇宙跳变，又被当作样本外起点；RankIC、Top20% 命中率和等权基准并非完全同口径。

影响：当前失败结论仍然成立，但这些历史估值和 OOS 结果不能成为未来“通过”的充分证据；2022 年以后也已被反复用于迭代，不再是干净留出集。

处理：保存真实发布时间、抓取时间和源版本；使用交易日历感知的 as-of merge；分别报告共同存续宇宙、当时有效宇宙和分类调整前后结果；未来晋级只认 2026-07-12 后的真实前推或全新未参与调参的数据。

#### 11. 审计覆盖率检查不完整

task brief 审计显示 97 项、0 error、0 warning，但它只检查“现有 brief 是否合规”，没有检查“每个研究版本是否都有 brief”。活动 brief 最新只覆盖到 V4.71，V4.72—V5.35 没有形成同等粒度的任务登记；实验哈希账本则只有 2 条真正的前推预登记。

影响：审计通过不能证明研究版本链完整，后续选择自由度无法从 brief 和实验账本中完整还原。

处理：每个可影响结论的研究版本必须同时存在 task brief、实验登记、changelog 和输出 manifest；缺一项即审计失败。

#### 12. 关键账本和多数报告直接覆盖

V5.25 与 V5.27 直接重写 CSV，没有临时文件、`fsync`、原子替换或进程锁。并发运行可能丢更新，进程中断可能留下半截账本。多数报告也采用直接覆盖，风险较低但同样可能产生半成品快照。

处理：关键账本采用追加式日志或临时文件 + `os.replace` + 文件锁；保留前一版本和哈希。报告输出也应先写临时目录，再整体切换。

#### 13. Dashboard 没把数据质量完整交给用户

数据构建器已经输出 `generated_at` 和 `data_quality_warnings`，前端却没有展示；“刷新”按钮只执行浏览器重载，不会刷新数据源或重建 JSON。600px 以下还隐藏数据截止日、估值快照日以及 PE、PB、股息率列，移动端只剩综合分数。

影响：用户可能把旧快照或降级数据误认为实时结果，移动端又看不到分数背后的基础指标。

处理：首屏固定显示生成时间、各数据源截止日、滞后天数和质量警告；明确“页面重载不等于数据刷新”；移动端至少保留估值日期和 PE/PB/股息率；对外使用前补充“不构成投资建议、历史不代表未来、数据可能延迟”的显式声明。

### P2：中期治理问题

#### 14. 数据许可与来源登记尚未闭环

项目已经主动标记公开申万估值源仍需口径、发布时间和授权源交叉审计，这是优点；但 `source_license_registry` 仍是 `planned_not_acquired`，没有可直接审计的数据使用登记。

处理：为每个当前主线源记录来源、接口、允许用途、保存范围、更新时间、失败降级方式和再分发边界。

#### 15. Dashboard 构建可通过，但存在便携性与体积问题

生成的 `dashboard_data.json` 写入了本机 `E:\投资\...` 绝对路径；构建后 CSS 约 838 KB、JS 约 426 KB。当前本机使用无碍，换目录或发布时不够干净。

处理：前端只接收相对证据路径或稳定 ID；发布前做静态资源拆分和缓存。

#### 16. 历史脚本存在较多宽泛异常捕获

多个旧版本脚本使用 `except Exception` 或空处理，容易把数据源变化、字段变化或编码错误降级成空结果。当前主线虽然有硬门禁，但历史研究结果的失败语义并不完全一致。

处理：新主线统一异常类型、失败代码和“保留最近合格快照”策略；不要求回改全部旧版本。

## 八、最终判断

项目最有价值的成果，不是 V4.70 的 100 分，也不是某一批候选行业，而是已经形成了一套会承认“没有证据”的研究框架。当前最大的风险不是程序跑不起来，而是文档、版本和审计名称让人误以为“工程完成”接近“策略有效”。
